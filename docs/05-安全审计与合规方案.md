# 安全审计与合规方案

## 一、安全审计方案

### 1.1 审计范围

| 审计层面 | 审计内容 | 优先级 | 推荐审计机构 |
|---------|---------|--------|------------|
| 链代码（Go） | Cosmos SDK 自定义模块 (`x/energy`, `x/oracle`, `x/identity`, `x/audit`)、EVM 集成逻辑、创世参数 | P0 | Trail of Bits, Halborn, Oak Security |
| 智能合约（Solidity） | RWA 全部合约（资产 NFT、收益权 Token、收益分配、预言机、DEX） | P0 | CertiK, OpenZeppelin, Peckshield, SlowMist |
| 基础设施 | RPC 端点安全、节点配置、密钥管理、网络架构 | P1 | 专业渗透测试团队 |
| 前端 DApp | XSS、CSRF、钱包签名安全、供应链攻击 | P2 | 安全团队 |

### 1.2 链代码审计重点

**Cosmos SDK 自定义模块审计要点：**

```
审计检查清单（Go 代码）：
──────────────────────
□ 消息验证（ValidateBasic）
  - 所有 Msg 类型是否正确实现了 ValidateBasic()
  - 是否存在整数溢出/下溢风险
  - 地址格式是否正确校验

□ Keeper 逻辑
  - Store 读写是否有权限校验
  - 状态变更是否原子化（失败时完整回滚）
  - 是否存在 panic 可能导致链停止

□ 创世逻辑（Genesis Import/Export）
  - Genesis 导入是否正确验证数据完整性
  - 导出/导入是否可逆（不丢失数据）

□ EVM 集成
  - Cosmos ↔ EVM 地址转换是否双向一致
  - EVM 交易的 Gas 计量是否与 Cosmos 一致
  - 预编译合约是否有权限边界问题

□ 共识安全
  - BeginBlock/EndBlock 逻辑是否确定性（所有节点一致）
  - 是否存在非确定性操作（随机数、时间精度、浮点运算）

□ IBC 安全
  - IBC 数据包处理是否正确
  - 超时处理是否完整
  - 中继器不可信假设是否成立
```

### 1.3 智能合约审计重点

```
审计检查清单（Solidity 合约）：
──────────────────────────
□ 重入攻击（Reentrancy）
  - RevenueDistributor.claim() 是否用了 ReentrancyGuard
  - 外部调用前是否已更新状态（Checks-Effects-Interactions 模式）

□ 整数溢出
  - Solidity 0.8.x 自带溢出检查，但仍需检查 unchecked 块
  - 18 位精度运算是否有精度丢失

□ 访问控制
  - 角色权限是否最小化
  - 初始角色分配是否正确
  - 是否存在未授权的特权函数

□ 快照机制安全
  - RevenueToken 的快照逻辑是否可以被操纵
  - 闪电贷攻击：攻击者是否能在同一交易中购买 Token → 快照 → 领取分红 → 卖出 Token

□ 预言机操纵
  - 喂价数据是否有范围检查（异常值过滤）
  - 单一喂价员被攻破的影响范围
  - 数据延迟/过期的处理是否安全

□ DEX 安全
  - 流动性池创建/销毁的边界条件
  - 价格操纵攻击（三明治攻击、闪电贷套利）
  - 首次添加流动性时的 LP Token 计算

□ 可升级性
  - 合约是否使用代理模式（建议初期不使用，降低复杂度）
  - 合约暂停/紧急关停机制是否可靠

□ Gas 优化
  - 存储布局是否紧凑
  - 循环是否有 Gas 上限（防止 DoS）
  - 批量操作是否有最大数量限制
```

### 1.4 审计流程

```
审计标准流程（约 6-8 周）：
─────────────────────────
第 1 周: 启动会议
  ├── 交付代码快照（特定 Git commit hash）
  ├── 提供架构文档和设计说明
  ├── 明确审计范围和时间线
  └── 确认沟通渠道和 POC

第 2-4 周: 审计执行
  ├── 审计师人工代码审查
  ├── 自动化工具扫描（Slither, Mythril, Semgrep）
  ├── 编写 PoC（Proof of Concept）复现漏洞
  └── 分类漏洞严重程度

第 5 周: 初步报告
  ├── 交付初步审计报告
  ├── 项目方审查并开始修复
  └── 对发现的问题进行答疑

第 6-7 周: 修复与复审
  ├── 项目方提交修复代码
  ├── 审计师验证修复是否有效
  └── 确认没有引入新问题

第 8 周: 最终报告
  ├── 交付最终审计报告（含修复确认）
  ├── 发布公开版本（去除敏感信息）
  └── 报告上传到官网/GitHub
```

### 1.5 漏洞严重等级定义

| 等级 | 定义 | 修复要求 | 示例 |
|------|------|---------|------|
| 严重（Critical） | 可直接导致资金损失 | 必须修复后才能上线 | 重入攻击导致资金被提空 |
| 高（High） | 可导致功能异常或间接资金风险 | 必须修复 | 权限绕过、逻辑错误 |
| 中（Medium） | 不影响资金但影响正常使用 | 应该修复 | Gas 优化问题、边界条件 |
| 低（Low） | 代码质量或最佳实践 | 建议修复 | 命名不规范、缺少事件日志 |
| 信息（Info） | 非安全问题的改进建议 | 可选 | 文档补充、代码可读性 |

| 机构 | 擅长领域 | 审计周期 | 参考报价（2026） | 报告质量 |
|------|---------|---------|----------------|---------|

## 二、持续安全措施

### 2.1 Bug Bounty 计划

上线 Immunefi 平台发布 Bug Bounty：

| 漏洞等级 | 赏金范围 | 适用范围 |
|---------|---------|---------|
| Critical | $10,000 - $50,000 | 可直接导致资金损失 |
| High | $5,000 - $10,000 | 权限绕过、逻辑错误 |
| Medium | $1,000 - $5,000 | 功能异常 |
| Low | $200 - $1,000 | 代码质量 |

### 2.2 多签国库

使用 Gnosis Safe（Safe）部署在 EnergyChain 上：

```
多签配置建议：
─────────────
钱包 1（项目方基金会）：     3/5 多签
钱包 2（团队 Token 管理）：   2/3 多签
钱包 3（应急暂停权限）：      2/3 多签

签名人应分布在不同地理位置、使用不同类型的签名设备（硬件钱包 + 软件钱包混合）
```

### 2.3 紧急响应预案

```
安全事件分级响应流程：
────────────────────
P0（资金正在被窃取）：
  1. 立即暂停受影响合约（Pausable.pause()）
  2. 通知所有 CEX 暂停 ECY/RWA 充提
  3. 分析攻击路径
  4. 修复并部署新合约（如需要）
  5. 发布事后分析报告（Post-Mortem）

P1（发现可利用漏洞但未被利用）：
  1. 评估漏洞严重性和可利用性
  2. 暂停相关功能
  3. 72 小时内完成修复
  4. 可选：奖励白帽报告者

P2（安全改进建议）：
  1. 评估并排入下一个开发周期
  2. 2 周内修复
```

### 2.4 合约升级策略

初期建议**不使用代理模式**（Proxy Pattern），原因：
- 降低审计复杂度
- 减少代理合约自身的安全风险
- 合约逻辑相对简单，不需要频繁升级

如需升级，采用**合约迁移**策略：
1. 部署新合约
2. 将状态从旧合约迁移到新合约（或引用旧合约状态）
3. 更新所有引用地址
4. 旧合约设为只读（暂停写入）

## 三、法律合规方案

### 3.1 法律实体架构

```
                    ┌─────────────────────┐
                    │  EnergyChain        │
                    │  Foundation Ltd     │  ← 新加坡基金会
                    │  (项目治理与代币发行) │     或 BVI/开曼
                    └─────────┬───────────┘
                              │
                ┌─────────────┼─────────────┐
                │             │             │
        ┌───────▼──────┐ ┌───▼──────┐ ┌───▼──────────┐
        │ 技术开发公司  │ │ 运营公司  │ │ RWA SPV      │
        │ (开发团队)   │ │ (社区运营)│ │ (资产持有)    │
        └──────────────┘ └──────────┘ └──────────────┘
```

**实体选择建议**：

| 实体 | 建议注册地 | 用途 | 理由 |
|------|-----------|------|------|
| 基金会 | 新加坡 | Token 发行主体、生态治理 | 加密友好、法律框架成熟 |
| 技术公司 | 中国/新加坡 | 技术开发 | 团队所在地 |
| 运营公司 | 新加坡/香港 | 社区运营、CEX 对接 | 国际化运营 |
| RWA SPV | 开曼/BVI | 持有能源资产权益 | 资产隔离、税务优化 |

### 3.2 Token 法律分类

ECY 和 RWA Token 的法律定性至关重要，不同定性意味着完全不同的合规要求：

**ECY（主网代币）**

| 测试项 | 分析 |
|--------|------|
| Howey Test | ECY 用于 Gas 支付和链安全质押，具有明确的功能用途，倾向于定性为 **功能型代币（Utility Token）** |
| 新加坡 MAS | 可能被归为 **Digital Payment Token**，适用 PSA 法案 |
| 香港 SFC | 如果仅用于 Gas/治理，可能不被视为证券 |
| 建议 | 获取法律意见书确认非证券定性 |

**RWA Token（收益权代币）**

| 测试项 | 分析 |
|--------|------|
| Howey Test | 代表电站未来收益权，持有人期望通过他人努力获利，**极可能被认定为证券（Security Token）** |
| 新加坡 MAS | 可能被归为 **Capital Markets Product**，需要 CMS 牌照 |
| 香港 SFC | 可能被视为 **Security Token**，需要 Type 1/7/9 牌照 |
| 建议 | 必须获取法律意见书；考虑双 Token 架构降低合规风险 |

### 3.3 双 Token 合规架构

为降低 RWA Token 的合规风险，建议采用双 Token 架构：

```
Token A: ECY（功能型代币）
──────────────────────────
  用途：Gas 支付、链安全质押、治理投票
  法律定性：Utility Token
  上 CEX：无限制
  转让限制：无

Token B: 收益权 Token（如 QHPV）
──────────────────────────
  方案 B1: 证券型 Token（机构版）
    - 仅面向合格投资者
    - 合约内置白名单（KYC/AML）
    - 通过持牌交易所或 STO 平台发行
    - 转让受限

  方案 B2: 收益分享 Token（零售版）
    - 法律架构上不构成证券
    - 代表"收益分享权"而非"资产所有权"
    - 无转让限制，可上普通 CEX
    - 通过法律实体和用户协议实现合规
    - 类似：项目分红、积分返利模式
```

### 3.4 法律意见书所需内容

需要律师事务所出具的法律意见书应包含：

1. **Token 性质分析**
   - ECY 在目标司法管辖区（新加坡、香港、美国）的法律定性
   - RWA Token 是否构成证券/集合投资计划/支付代币
   
2. **合规建议**
   - 所需牌照和资质
   - KYC/AML 要求
   - 投资者适当性要求
   
3. **风险提示**
   - 监管不确定性
   - 跨境法律冲突
   
4. **结论性意见**
   - 明确 Token 可以/不可以做什么
   - CEX 上币的法律可行性

**推荐律所**：
- Dentons（大成）—— Cosmos 生态项目经验
- Simmons & Simmons —— 亚洲加密合规
- Clifford Chance —— 国际化，CEX 认可度高
- 中伦 / 金杜 —— 中国法律+跨境
- Rajah & Tann —— 新加坡本地

### 3.5 KYC/AML 集成方案

如果 RWA Token 需要 KYC：

**链上方案（使用 Identity 模块）**：
- 激活 `chain/x/identity/` 模块
- KYC 验证通过后，将地址注册到链上 Identity 模块
- 合约在 transfer 时检查 Identity 模块
- 优点：去中心化；缺点：隐私差、开发量大

**链下方案（推荐 MVP）**：
- 使用第三方 KYC 服务商（Sumsub、Jumio、Onfido）
- KYC 通过后，管理员将地址加入合约白名单
- 优点：开发简单、隐私保护；缺点：中心化

## 四、做市商对接方案

### 4.1 做市商的作用

| 作用 | 说明 |
|------|------|
| 提供流动性 | 在 CEX 挂买卖单，确保用户能随时交易 |
| 缩小价差 | 减小买一价和卖一价之间的差距 |
| 稳定价格 | 防止大额交易导致价格剧烈波动 |
| CEX 要求 | 多数一线所要求项目方自行安排做市商 |

### 4.2 做市商模式

**模式 A：借币做市（Loan-Based）**

```
项目方借出 Token 给做市商：
  ECY 借出量：5,000,000 ECY（流动性储备的 10%）
  借出期限：12 个月
  做市承诺：
    - 维持买卖价差 < 2%
    - 维持最小挂单深度 > $50,000
    - 每日交易量 > $100,000
  费用：无前期费用，到期归还等量 Token（或按约定差价结算）
```

**模式 B：付费做市（Retainer-Based）**

```
项目方支付固定月费：
  月费：$10,000 - $30,000
  做市承诺：同上
  无需借出 Token
  适合：不想稀释 Token 的项目
```

**模式 C：收益分享（Revenue-Share）**

```
做市商自投资金，利润分成：
  做市商自投入 USDT 作为对手方
  利润分成：做市商 70% / 项目方 30%
  适合：市场看好的项目，做市商愿意承担风险
```

### 4.3 主流做市商清单

| 做市商 | 擅长市场 | 最低合作门槛 | 联系渠道 |
|--------|---------|-------------|---------|
| Wintermute | CEX + DEX | 中型项目+ | 官网申请 |
| GSR | 机构级 | 大型项目 | 推荐制 |
| Amber Group | 亚洲市场 | 中小型项目 | 商务对接 |
| Jump Crypto | 顶级 | 大型项目 | 推荐制 |
| DWF Labs | 积极做市 | 中小型项目 | 官网申请 |
| Keyrock | 欧洲市场 | 中型项目 | 官网申请 |
| Flowdesk | CEX + 合规 | 中小型项目 | 官网申请 |

**建议方案**：
- 优先联系 Amber Group（亚洲市场经验好）和 DWF Labs（合作门槛较低）
- 采用借币做市模式，降低前期现金支出
- ECY 上所时安排做市商，RWA Token 可后续追加

### 4.4 做市商技术对接

做市商需要的技术信息：

```
做市商技术对接清单：
────────────────────
□ Chain ID 和网络信息
□ RPC 端点（低延迟、高可用）
□ Token 合约地址（RWA Token 的 ERC-20 地址）
□ 交易手续费结构（Gas 费）
□ 区块时间和终局性
□ 做市商账户充值地址
□ Token 精度（decimals）
□ 做市 API（如果 DEX，需要合约 ABI）
□ 历史价格数据 feed
```
