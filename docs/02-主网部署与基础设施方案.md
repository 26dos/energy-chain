## 一、主网部署方案

### 1.1 主网参数

| 参数 | 测试网（当前） | 主网（目标） |
|------|-------------|-------------|
| Chain ID | 9001 | energychain_1-1 |
| 出块时间 | ~1s（timeout_commit=1s） | ~3s（timeout_commit=3s） |
| 最大验证人数 | 1 | 100 |
| 最小质押量 | 1,000 ECY | 10,000 ECY |
| 解绑期 | 测试短周期 | 21 天（1,814,400s） |
| 最小 Gas 价格 | 10 Gwei | 10 Gwei |
| Block Max Gas | 10,000,000 | 30,000,000 |
| 治理投票期 | 30s（测试） | 7 天（604,800s） |

### 1.2 创世仪式（Genesis Ceremony）流程

创世仪式是多方参与生成主网 genesis.json 的标准化流程。

**阶段一：准备期（T-14 天）**

1. 项目方发布主网二进制版本（Release Tag），如 `v1.0.0`
2. 公布创世参数草案（代币分配、模块参数）
3. 验证人候选人下载二进制，测试兼容性
4. 收集验证人候选人信息：
   - 运营实体名称
   - 节点公钥（`energychaind tendermint show-validator`）
   - 自质押金额
   - commission rate
   - 安全联系方式

**阶段二：创世 Gentx 收集（T-7 天）**

```bash
# 每个验证人在本地执行
energychaind init <moniker> --chain-id energychain_1-1

# 导入项目方提供的预创世文件（含代币分配）
cp pre-genesis.json ~/.energychaind/config/genesis.json

# 添加自己的验证人密钥
energychaind keys add <validator_name> --keyring-backend file --algo eth_secp256k1

# 创建 gentx（创世质押交易）
energychaind genesis gentx <validator_name> <stake_amount>uecy \
  --chain-id energychain_1-1 \
  --commission-rate 0.10 \
  --commission-max-rate 0.20 \
  --commission-max-change-rate 0.01 \
  --min-self-delegation 1 \
  --gas-prices 10000000000uecy \
  --keyring-backend file

# 提交 gentx 文件到 GitHub 仓库
# 文件位于 ~/.energychaind/config/gentx/gentx-*.json
```

**阶段三：创世文件合并与验证（T-3 天）**

```bash
# 项目方收集所有 gentx，合并到创世文件
energychaind genesis collect-gentxs

# 验证创世文件
energychaind genesis validate-genesis

# 发布最终 genesis.json（SHA256 校验和）
sha256sum ~/.energychaind/config/genesis.json
```

**阶段四：主网启动（T=0）**

1. 所有验证人下载最终 genesis.json
2. 配置 seeds / persistent_peers
3. 约定统一启动时间（genesis_time 字段）
4. 所有节点在 genesis_time 后自动开始出块

### 1.3 节点架构

主网推荐采用 **Sentry 节点架构**，保护验证人不被直接暴露：

```
                    ┌─────────────┐
                    │  用户/CEX    │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼─────┐ ┌───▼─────┐ ┌───▼─────┐
        │ RPC 节点 1 │ │RPC 节点2│ │RPC 节点3│  ← 公共 RPC（负载均衡）
        │(全节点)    │ │(全节点) │ │(Archive)│
        └─────┬─────┘ └───┬─────┘ └───┬─────┘
              │            │            │
              └────────────┼────────────┘
                           │ 私有网络
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼─────┐ ┌───▼─────┐ ┌───▼─────┐
        │ Sentry A   │ │Sentry B│ │Sentry C │  ← 哨兵节点（中继层）
        │(全节点)    │ │(全节点) │ │(全节点) │
        └─────┬─────┘ └───┬─────┘ └───┬─────┘
              │            │            │
              └────────────┼────────────┘
                           │ 隔离网络
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼─────┐ ┌───▼─────┐ ┌───▼─────┐
        │Validator 1│ │Validator2│ │  ... N  │  ← 验证人节点（不对外）
        └───────────┘ └─────────┘ └─────────┘
```

**角色说明：**

| 节点类型 | 数量 | 对外暴露 | 职责 |
|---------|------|---------|------|
| 验证人节点 | 10-100 | 否 | 参与共识出块，不对外暴露 P2P 或 RPC |
| 哨兵节点 | ≥ 2/验证人 | 仅 P2P | 中继交易和区块，保护验证人 |
| 全节点（RPC） | ≥ 3 | JSON-RPC + REST + gRPC | 接收用户交易，提供查询 API |
| Archive 节点 | ≥ 1 | JSON-RPC | 保存全量历史状态，供 CEX 查历史余额 |
| Seed 节点 | ≥ 2 | 仅 P2P | 帮助新节点发现网络中的其他节点 |

### 1.4 验证人硬件要求

| 节点类型 | CPU | 内存 | 磁盘 | 网络 |
|---------|-----|------|------|------|
| 验证人 | 8 核+ | 32 GB+ | 500 GB NVMe SSD | 100 Mbps+ |
| 全节点/哨兵 | 4 核+ | 16 GB+ | 500 GB SSD | 100 Mbps+ |
| Archive 节点 | 8 核+ | 64 GB+ | 2 TB+ NVMe SSD | 200 Mbps+ |
| Seed 节点 | 2 核 | 8 GB | 200 GB SSD | 100 Mbps+ |

### 1.5 验证人加入流程（主网已启动后）

```bash
# 1. 安装二进制
git clone https://github.com/energy-chain/energy-chain.git
cd energy-chain/chain && go build -o energychaind ./cmd/energychaind

# 2. 初始化节点
energychaind init <moniker> --chain-id energychain_1-1

# 3. 下载主网 genesis.json
wget -O ~/.energychaind/config/genesis.json https://raw.githubusercontent.com/energy-chain/mainnet/main/genesis.json

# 4. 配置 seeds 和 persistent_peers
# 编辑 ~/.energychaind/config/config.toml
# seeds = "node_id1@seed1.energychain.io:26656,node_id2@seed2.energychain.io:26656"

# 5. (可选) 使用 State Sync 快速同步
# 编辑 config.toml 的 [statesync] 段
# enable = true
# rpc_servers = "https://rpc1.energychain.io:443,https://rpc2.energychain.io:443"
# trust_height = <recent_height>
# trust_hash = "<block_hash_at_trust_height>"

# 6. 启动节点同步
energychaind start

# 7. 等待同步完成后，创建验证人
energychaind tx staking create-validator \
  --amount 10000000000000000000000uecy \
  --pubkey $(energychaind tendermint show-validator) \
  --moniker "<moniker>" \
  --chain-id energychain_1-1 \
  --commission-rate 0.10 \
  --commission-max-rate 0.20 \
  --commission-max-change-rate 0.01 \
  --min-self-delegation 1 \
  --gas-prices 10000000000uecy \
  --from <key_name> \
  --keyring-backend file
```

## 二、Cosmovisor 链升级机制

### 2.1 Cosmovisor 简介

Cosmovisor 是 Cosmos SDK 官方的进程管理工具，用于自动化链升级：
- 监控链上治理升级提案
- 在指定区块高度自动切换到新版本二进制
- 零停机升级（整个网络在同一区块高度统一切换）

### 2.2 Cosmovisor 目录结构

```
~/.energychaind/
├── cosmovisor/
│   ├── genesis/
│   │   └── bin/
│   │       └── energychaind        ← 初始版本二进制
│   ├── upgrades/
│   │   ├── v1.1.0/
│   │   │   └── bin/
│   │   │       └── energychaind    ← v1.1.0 升级二进制
│   │   └── v1.2.0/
│   │       └── bin/
│   │           └── energychaind    ← v1.2.0 升级二进制
│   └── current -> genesis/         ← 符号链接，指向当前版本
├── config/
│   ├── genesis.json
│   ├── config.toml
│   └── app.toml
└── data/
```

### 2.3 安装与配置

```bash
# 安装 Cosmovisor
go install cosmossdk.io/tools/cosmovisor/cmd/cosmovisor@latest

# 设置环境变量
export DAEMON_NAME=energychaind
export DAEMON_HOME=$HOME/.energychaind
export DAEMON_ALLOW_DOWNLOAD_BINARIES=false  # 安全起见，不允许自动下载
export DAEMON_RESTART_AFTER_UPGRADE=true
export DAEMON_LOG_BUFFER_SIZE=512
export UNSAFE_SKIP_BACKUP=false  # 升级前自动备份

# 初始化 Cosmovisor 目录
cosmovisor init $(which energychaind)

# 用 Cosmovisor 启动节点（替代直接启动 energychaind）
cosmovisor run start \
  --pruning nothing \
  --minimum-gas-prices=10000000000uecy \
  --evm.min-tip=0 \
  --json-rpc.api eth,txpool,personal,net,debug,web3
```

### 2.4 升级流程

**第一步：开发新版本**

在 `chain/app/upgrades.go` 中注册升级处理器：

```go
func (app *App) RegisterUpgradeHandlers() {
    app.UpgradeKeeper.SetUpgradeHandler(
        "v1.1.0",
        func(ctx context.Context, plan upgradetypes.Plan, fromVM module.VersionMap) (module.VersionMap, error) {
            // 升级逻辑：参数修改、数据迁移等
            return app.ModuleManager.RunMigrations(ctx, app.configurator, fromVM)
        },
    )
}
```

**第二步：发起链上治理提案**

```bash
energychaind tx gov submit-proposal software-upgrade v1.1.0 \
  --title "Upgrade to v1.1.0" \
  --description "Enable RWA module and Oracle module" \
  --upgrade-height <target_block_height> \
  --deposit 10000000000000000000000uecy \
  --from <proposer_key> \
  --chain-id energychain_1-1 \
  --gas-prices 10000000000uecy
```

**第三步：验证人投票**

```bash
energychaind tx gov vote <proposal_id> yes \
  --from <validator_key> \
  --chain-id energychain_1-1 \
  --gas-prices 10000000000uecy
```

**第四步：准备新版本二进制**

```bash
# 验证人在升级高度到达前准备新二进制
cosmovisor add-upgrade v1.1.0 /path/to/new/energychaind
```

**第五步：自动切换**

当链到达 upgrade-height 时：
1. 所有节点暂停出块
2. Cosmovisor 检测到升级计划
3. 自动切换到 `upgrades/v1.1.0/bin/energychaind`
4. 执行升级处理器中的迁移逻辑
5. 继续出块

### 2.5 升级回滚预案

如果升级出现问题：

```bash
# 停止 Cosmovisor
pkill cosmovisor

# 恢复备份（如果 UNSAFE_SKIP_BACKUP=false，升级前有自动备份）
cp -r ~/.energychaind/data-backup-<date> ~/.energychaind/data

# 手动将 current 符号链接指回旧版本
ln -sf ~/.energychaind/cosmovisor/genesis ~/.energychaind/cosmovisor/current

# 重启
cosmovisor run start ...
```

## 三、基础设施部署方案

### 3.1 部署区域规划

| 区域 | 节点类型 | 服务商建议 | 用途 |
|------|---------|-----------|------|
| 亚洲（香港/新加坡） | 2x 全节点 RPC + 1x Archive | AWS / Alibaba Cloud | 主要用户市场 |
| 欧洲（法兰克福） | 1x 全节点 RPC + 1x Seed | AWS / Hetzner | CEX 节点中继 |
| 北美（弗吉尼亚） | 1x 全节点 RPC + 1x Seed | AWS / Digital Ocean | CEX 备用 |

### 3.2 RPC 负载均衡

使用 Nginx 作为 RPC 入口反向代理：

```
用户请求 → rpc.energychain.io → Nginx LB → 后端 RPC 节点池
```

**Nginx 配置要点：**

```nginx
upstream energychain_rpc {
    least_conn;
    server rpc-node-1:8545 max_fails=3 fail_timeout=30s;
    server rpc-node-2:8545 max_fails=3 fail_timeout=30s;
    server rpc-node-3:8545 backup;
}

upstream energychain_ws {
    ip_hash;  # WebSocket 需要会话保持
    server rpc-node-1:8546;
    server rpc-node-2:8546;
}

server {
    listen 443 ssl;
    server_name rpc.energychain.io;

    # JSON-RPC (HTTP)
    location / {
        proxy_pass http://energychain_rpc;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;

        # Rate limiting
        limit_req zone=rpc burst=50 nodelay;
    }

    # WebSocket
    location /ws {
        proxy_pass http://energychain_ws;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 86400s;
    }
}
```

### 3.3 State Sync 配置

允许新节点快速同步（几分钟 vs 几天）：

在 RPC 节点的 `config.toml` 中启用快照：

```toml
[statesync]
# 每 1000 个区块生成一次快照
snapshot-interval = 1000
snapshot-keep-recent = 3
```

新节点加入时使用 State Sync：

```toml
[statesync]
enable = true
rpc_servers = "https://rpc1.energychain.io:443,https://rpc2.energychain.io:443"
trust_height = 100000
trust_hash = "ABCDEF..."
trust_period = "168h"
```

### 3.4 监控方案

**Prometheus + Grafana 监控栈**

节点已内置 Prometheus metrics 端点（端口 26660）：

关键监控指标：

| 指标 | 含义 | 告警阈值 |
|------|------|---------|
| `cometbft_consensus_height` | 当前区块高度 | 5 分钟未增长 |
| `cometbft_consensus_validators` | 活跃验证人数 | < 最小安全阈值 |
| `cometbft_consensus_missing_validators` | 缺席验证人数 | > 总量 1/3 |
| `cometbft_consensus_rounds` | 当前轮次 | > 1（出块异常） |
| `cometbft_mempool_size` | 内存池交易数 | > 5000（拥堵） |
| `cometbft_p2p_peers` | P2P 连接数 | < 3 |
| `process_resident_memory_bytes` | 进程内存 | > 28 GB |
| `process_cpu_seconds_total` | CPU 使用 | 持续 > 90% |

**Grafana Dashboard 配置：**
- 使用社区现成的 Cosmos SDK Dashboard（ID: 11036）
- 自定义 EVM 指标面板（RPC 请求量、Gas 使用率）

### 3.5 备份策略

```bash
# 每日自动备份脚本
#!/bin/bash
BACKUP_DIR="/backup/energychain/$(date +%Y%m%d)"
mkdir -p $BACKUP_DIR

# 备份数据目录
tar czf $BACKUP_DIR/data.tar.gz ~/.energychaind/data/

# 备份配置
cp ~/.energychaind/config/genesis.json $BACKUP_DIR/
cp ~/.energychaind/config/config.toml $BACKUP_DIR/
cp ~/.energychaind/config/app.toml $BACKUP_DIR/

# 备份验证人私钥（加密）
gpg --encrypt --recipient admin@energychain.io \
  ~/.energychaind/config/priv_validator_key.json \
  -o $BACKUP_DIR/priv_validator_key.json.gpg

#  保留最近 7 天备份
find/backup/energychain/ -maxdepth 1 -mtime +7 -exec rm -rf {} \;
```
