# 能源公链（EnergyChain）· 基础实现与架构

> 基于 **Cosmos SDK v0.54 + Cosmos EVM + CometBFT v0.39** 构建的能源行业公链。
> 采用国际通用加密算法（非国密），与 Ethereum 生态完全兼容。

## 三、共识机制

### 3.1 CometBFT（Tendermint）BFT 共识

EnergyChain 使用 **CometBFT v0.39** 作为共识引擎，采用 **BFT（拜占庭容错）+ PoS（权益证明）** 机制。

**核心特性**：
- **即时最终性**：区块一旦提交，不可回滚（无需等待确认数）
- **容错能力**：可容忍最多 1/3 的验证者恶意或离线
- **确定性出块**：每个高度只有一个合法区块

### 3.2 出块流程

```
Height = N

┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  1. Propose   │ ──► │  2. Prevote   │ ──► │ 3. Precommit │ ──► │  4. Commit   │
│  提议者打包    │     │  各节点验证    │     │  收集 2/3+    │     │  写入区块     │
│  区块并广播    │     │  投票赞成/反对  │     │  预提交投票    │     │  进入下一高度  │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

1. **Propose**：按轮换规则选出提议者，从交易池取交易构造区块
2. **Prevote**：验证者验证区块有效性（交易、状态根、Gas），广播投票
3. **Precommit**：收到 >2/3 的 Prevote 后，广播 Precommit 投票
4. **Commit**：收到 >2/3 的 Precommit 后，区块正式提交，所有节点写入存储

### 3.3 当前链配置参数

| 参数 | 当前值 | 说明 |
|------|--------|------|
| 出块提议超时 | 2 秒 | `timeout_propose = "2s"` |
| 出块提交超时 | 1 秒 | `timeout_commit = "1s"` |
| 实际出块间隔 | ~1-2 秒 | 测试网快速出块 |
| 区块最大 Gas | 10,000,000 | `consensus.params.block.max_gas` |
| 验证者数量 | 1（测试网） | 可通过质押增加 |

### 3.4 提议者选举

CometBFT 采用**加权轮换（Weighted Round-Robin）**算法选举提议者：
- 按验证者的投票权（Voting Power，即质押量）加权
- 质押越多，被选为提议者的频率越高
- 算法确定性，所有节点计算结果一致

### 3.5 区块结构

CometBFT 区块由以下部分组成：

```
Block:
  Header:
    chain_id            # 链标识 ("9001")
    height              # 区块高度
    time                # 时间戳
    last_block_id       # 上一区块哈希
    data_hash           # 交易数据哈希
    validators_hash     # 验证者集合哈希
    next_validators_hash
    consensus_hash      # 共识参数哈希
    app_hash            # 应用状态根（Cosmos SDK 计算）
    last_results_hash   # 上一区块交易执行结果哈希
    proposer_address    # 提议者地址
  Data:
    txs[]               # 交易列表（包括 Cosmos 原生交易和 EVM 交易）
  Evidence:
    evidence[]          # 不良行为证据（双签等）
  LastCommit:
    signatures[]        # 上一区块的验证者签名集
```

## 四、节点类型与角色

### 4.1 节点分类

| 节点类型 | 角色 | 存储 | 参与共识 | 典型用途 |
|----------|------|------|----------|----------|
| **共识节点 (Validator)** | 出块、投票、验证 | 全状态 + 全区块 | 是 | 维护网络安全与最终性 |
| **全节点 (Full Node)** | 验证并转发区块/交易 | 全状态 + 全区块（可裁剪） | 否 | RPC 服务、轻量验证 |
| **归档节点 (Archive)** | 保留全部历史状态 | 全状态 + 全历史 | 否 | 区块浏览器、审计、历史查询 |
| **轻节点 (Light)** | 仅验证区块头 | 区块头 + Merkle 证明 | 否 | 移动端、资源受限环境 |

### 4.2 验证者管理

- **准入**：任何人通过 `tx staking create-validator` 并质押足够 ECY 即可成为验证者
- **活跃集**：按质押量排序取 Top-N 进入活跃验证者集合
- **委托**：持币者可通过 `tx staking delegate` 将 ECY 委托给验证者，共享出块收益
- **解绑期**：解绑需等待 21 天（可通过治理调整）
- **当前测试网**：1 个验证者，质押 1,000 ECY，投票权 1000

### 4.3 节点间通信

- **协议**：CometBFT P2P 协议
- **发现**：通过 `seeds` 和 `persistent_peers` 配置引导节点
- **广播**：区块和交易通过 gossip 协议在节点间传播
- **同步**：支持全量同步和快速同步（state sync）

## 五、加密算法（非国密）

### 5.1 算法选型与实现

| 用途 | 算法 | 具体实现 | 说明 |
|------|------|----------|------|
| **账户密钥** | `eth_secp256k1` | Cosmos EVM 的 `crypto/hd` 包 | 与 Ethereum 钱包完全兼容 |
| **签名/验签** | ECDSA over secp256k1 | libsecp256k1 | 交易签名、消息验证 |
| **地址派生** | Keccak-256 | 取公钥哈希后 20 字节 | EVM 地址格式 `0x...` |
| **交易/区块哈希** | Keccak-256 | 与 Ethereum 一致 | EVM 侧使用 |
| **状态树哈希** | SHA-256 | Cosmos SDK IAVL 树 | Cosmos 侧使用 |
| **Bech32 编码** | Bech32 | 前缀 `cosmos` | Cosmos 侧地址表示 |
| **HD 钱包路径** | BIP-44 | `m/44'/60'/0'/0/N` | coin_type=60，与 Ethereum 一致 |

### 5.2 双地址格式

EnergyChain 的每个账户同时拥有两种地址格式：

| 格式 | 前缀 | 用途 | 示例 |
|------|------|------|------|
| **EVM 地址** | `0x` | EVM 合约交互、MetaMask | `0xA2A8B87390F8F2D188242656BFb6852914073D06` |
| **Cosmos 地址** | `cosmos` | Cosmos 原生交易、Keplr | `cosmos1520gz...` |

两种地址由同一私钥派生，对应同一个账户和余额。

### 5.3 Bech32 地址前缀体系

| 类型 | 前缀 | 用途 |
|------|------|------|
| 账户地址 | `cosmos` | 普通用户账户 |
| 账户公钥 | `cosmospub` | 公钥展示 |
| 验证者操作地址 | `cosmosvaloper` | 验证者管理 |
| 验证者操作公钥 | `cosmosvaloperpub` | 验证者公钥 |
| 共识节点地址 | `cosmosvalcons` | 共识签名地址 |
| 共识节点公钥 | `cosmosvalconspub` | 共识公钥 |

### 5.4 不采用国密的理由

- **公链兼容性**：secp256k1 + Keccak-256 与 Ethereum 全生态（MetaMask、Hardhat、Ethers.js）无缝对接
- **钱包通用性**：用户可直接使用 MetaMask、Keplr 等主流钱包
- **工具链成熟**：国际算法有大量成熟实现和安全审计
- **开发者友好**：Solidity 合约开发者零迁移成本

## 六、Token 经济模型

### 6.1 原生代币 ECY

| 属性 | 值 |
|------|------|
| 名称 | Energy Chain Yield |
| 符号 | **ECY** |
| 最小单位 | **uecy**（micro ECY） |
| 精度 | 18 位（1 ECY = 10^18 uecy） |
| EVM 中表现 | 等同于 ETH 的角色（支付 Gas） |

### 6.2 代币元数据定义（Genesis 配置）

```json
{
  "denom_units": [
    {"denom": "uecy", "exponent": 0, "aliases": ["microecy"]},
    {"denom": "ecy", "exponent": 18}
  ],
  "base": "uecy",
  "display": "ecy",
  "name": "Energy Chain Yield",
  "symbol": "ECY"
}
```

### 6.3 代币角色

| 角色 | 说明 |
|------|------|
| **Gas 费** | 每笔 EVM/Cosmos 交易都需支付 ECY 作为 Gas 费 |
| **质押 (Staking)** | 验证者质押 ECY 获得出块权，委托者质押分享收益 |
| **出块奖励** | 新铸造的 ECY 按质押权重分配给验证者和委托者 |
| **治理** | 质押者参与链上提案投票（参数变更、升级等） |
| **跨链** | 通过 IBC 协议可转移到其他 Cosmos 生态链 |

### 6.4 创世分配（测试网）

| 账户 | 分配量 | 用途 |
|------|--------|------|
| validator | 100,000,000 ECY | 验证出块 + 合约部署 + 测试 |
| dev0 | 1,000 ECY | 开发测试 |
| validator 质押 | 1,000 ECY | 初始质押量 |

### 6.5 质押与奖励

- **质押**：验证者通过 `gentx` 或 `tx staking create-validator` 质押 ECY
- **委托**：持币者通过 `tx staking delegate` 委托给验证者
- **奖励来源**：
  - 区块铸币奖励（mint 模块）
  - 交易手续费（Gas 费）
- **分配规则**：验证者抽取佣金后，剩余按委托比例分配
- **解绑期**：解除质押或委托后，需等待解绑期（默认 21 天）

### 6.6 罚没 (Slashing)

| 罚没条件 | 说明 | 惩罚 |
|----------|------|------|
| **双签 (Double Sign)** | 同一高度对两个区块签名 | 扣减质押 5%，永久禁用 |
| **长时间离线** | 连续缺席超过阈值 | 扣减质押 0.01%，暂时禁用 |

## 七、Gas 消耗机制

### 7.1 EIP-1559 动态 Fee Market

EnergyChain 通过 `feemarket` 模块实现 **EIP-1559** 动态 Gas 定价：

| 参数 | 当前值 | 说明 |
|------|--------|------|
| `base_fee` | 10 Gwei（10,000,000,000 uecy） | 基础费用，根据区块利用率动态调整 |
| `min_gas_price` | 10 Gwei | 基础费用不会低于此值 |
| `base_fee_change_denominator` | 8 | 基础费用每块最多变化 1/8（12.5%） |
| `elasticity_multiplier` | 2 | 区块 Gas 弹性系数 |
| `min_gas_multiplier` | 0.5 | 最小 Gas 乘数 |

### 7.2 Gas 费计算

```
交易费 = Gas Used × Effective Gas Price
Effective Gas Price = Base Fee + Priority Fee (Tip)
```

**实际示例**（当前测试网）：

| 操作 | Gas Used | Gas Price | 手续费 |
|------|----------|-----------|--------|
| 单条数据存证 | ~238,000 | 11.25 Gwei | ~0.0027 ECY |
| 批量存证 (5 条) | ~1,200,000 | 11.25 Gwei | ~0.0135 ECY |
| 合约部署 | ~1,100,000 | 11.25 Gwei | ~0.0124 ECY |

### 7.3 Gas 费分配

- **Base Fee**：EIP-1559 机制下可用于销毁或进入社区池
- **Priority Fee (Tip)**：直接奖励给区块提议者
- **节点启动参数**：`--minimum-gas-prices=10000000000uecy --evm.min-tip=0`

### 7.4 区块 Gas 限制

| 参数 | 值 |
|------|------|
| 区块最大 Gas | 10,000,000 |
| 单笔交易 Gas | 用户自行设置 `gas_limit` |

## 八、EVM 兼容层

### 8.1 Cosmos EVM 模块

EnergyChain 通过 `github.com/cosmos/evm` 模块实现完整的 EVM 兼容：

| 维度 | 实现 |
|------|------|
| **字节码** | 完整支持 EVM bytecode，Solidity/Vyper 编译产物可直接部署 |
| **Opcode 集** | 与 Ethereum 完全一致（支持到 Cancun 版本） |
| **Gas 表** | 与 Ethereum 一致 |

### 9.3 溯源原理

```
【上链阶段】
  原始数据 ─── keccak256 (链下) ───► 32字节数据哈希
                                         │
                                         ▼
                               ┌──────────────────┐
                               │  合约存储          │
                               │  dataHash ← 指纹  │
                               │  submitter ← 谁    │
                               │  timestamp ← 何时  │
                               │  blockNumber       │
                               │  dataType ← 业务   │
                               └──────────────────┘

【验证阶段】
  待验证数据 ─── keccak256 ───► 重算哈希 ──► 链上查询比对
                                              │
                                        匹配 ─┼─ 不匹配
                                              │
                                     ✓ 未被篡改     ✗ 已篡改或未上链
                                     ✓ 时间可查
                                     ✓ 提交者可溯

【不可篡改性保障】
  · BFT 共识：>2/3 验证者确认才入块
  · 链式哈希：每个区块包含前一区块哈希
  · 智能合约：合约代码不可修改
  · 防重复：同一数据哈希只能存证一次
```

## 十六、安全注意事项

### 16.1 共识安全

| 风险 | 应对 |
|------|------|
| 共识逻辑 Bug | 使用 CometBFT 成熟实现，非自研共识 |
| 拜占庭容错边界（≤1/3） | 增加验证者数量和质押分散度 |
| 长程攻击 | CometBFT 弱主观性检查点 |
| Nothing-at-Stake | slashing 模块罚没双签 |

### 16.2 EVM 执行安全

| 风险 | 应对 |
|------|------|
| EVM 不一致 | 使用 go-ethereum 的 Cosmos 定制分支，经过充分测试 |
| Gas 计算错误 | 复用 Ethereum 标准 Gas 表 |
| 预编译实现错误 | 使用 Cosmos EVM 提供的审计过的预编译 |

### 16.3 运维安全

| 风险 | 应对 |
|------|------|
| 私钥泄露 | 分离共识签名密钥与质押/提款密钥 |
| 节点监控 | Prometheus 指标已启用 (`prometheus = true`) |
| RPC 暴露 | 生产环境应限制 `debug`/`personal` 命名空间 |
| CORS 配置 | 测试网已开启 (`enabled-unsafe-cors = true`)，生产环境需限制 |

### 16.4 测试网安全声明

当前配置为**测试网模式**：
- 助记词和私钥公开（仅用于测试）
- CORS 全开放
- 所有 RPC 命名空间开放
- 单验证者运行

生产环境部署需要：关闭 CORS、限制 RPC、多验证者、HSM 密钥管理、防火墙规则等。
