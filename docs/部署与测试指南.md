## 四、初始化并启动测试网

```bash
cd chain

# -y 覆盖旧数据，自动初始化并启动
bash scripts/local_node.sh -y
```

脚本自动完成：
1. 停止旧 energychaind 进程
2. 删除旧链数据 `~/.energychaind`
3. 创建 validator 和 dev0 两个账户
4. 初始化创世区块（ECY 代币、10 Gwei Gas 价格、EVM 参数）
5. 配置 2 秒出块、60M 区块 Gas 上限、20000 mempool
6. 启动节点

**启动后保持此终端运行**，另开终端执行后续操作。

### 节点端口

| 端口 | 协议 | 用途 |
|------|------|------|
| 8545 | EVM JSON-RPC (HTTP) | 合约部署、CLI 交互 |
| 8546 | EVM JSON-RPC (WebSocket) | 事件订阅 |
| 26657 | CometBFT RPC | Cosmos 侧查询 |
| 1317 | Cosmos REST API | Cosmos 侧 REST |
| 9090 | gRPC | Cosmos gRPC |

## 九、数据溯源验证与篡改检测

> 这是整个系统的**核心能力**：证明数据未被篡改，且确实在特定时间上链。

### 9.1 正常验证流程

先上链一条数据：

```bash
node bin/energy-cli.js attest \
  -d '{"station_id":"CS-HZ-001","order_id":"ORD-20260316-001","user_id":"U10086","kwh":35.6,"amount_yuan":28.48,"start":"2026-03-16T08:30:00","end":"2026-03-16T09:15:00"}' \
  -t "charging_order" \
  -m "杭州充电桩订单"
```

用**完全相同的原始数据**验证：

```bash
node bin/energy-cli.js verify \
  -d '{"station_id":"CS-HZ-001","order_id":"ORD-20260316-001","user_id":"U10086","kwh":35.6,"amount_yuan":28.48,"start":"2026-03-16T08:30:00","end":"2026-03-16T09:15:00"}'
```

预期输出：
```
✓ 验证通过 — 数据已上链，未被篡改

📋 链上存证记录:
  数据哈希: 0xabcdef...
  提 交 者: 0xA2A8B87390F8F2D188242656BFb6852914073D06
  上链时间: 2026-03-16T12:30:54.000Z
  区 块 号: 25
  业务类型: charging_order
  备    注: 杭州充电桩订单
```

### 9.2 篡改检测演示（核心！）

**场景 1：修改金额**

```bash
# 原始: amount_yuan = 28.48 → 篡改为 9999
node bin/energy-cli.js verify \
  -d '{"station_id":"CS-HZ-001","order_id":"ORD-20260316-001","user_id":"U10086","kwh":35.6,"amount_yuan":9999,"start":"2026-03-16T08:30:00","end":"2026-03-16T09:15:00"}'
```

预期输出：
```
✗ 验证失败 — 链上无此数据记录
```

**场景 2：修改用电量**

```bash
# 原始: kwh = 35.6 → 篡改为 100
node bin/energy-cli.js verify \
  -d '{"station_id":"CS-HZ-001","order_id":"ORD-20260316-001","user_id":"U10086","kwh":100,"amount_yuan":28.48,"start":"2026-03-16T08:30:00","end":"2026-03-16T09:15:00"}'
```

预期输出：`✗ 验证失败`

**场景 3：增加字段**

```bash
# 加了一个 "discount": true 字段
node bin/energy-cli.js verify \
  -d '{"station_id":"CS-HZ-001","order_id":"ORD-20260316-001","user_id":"U10086","kwh":35.6,"amount_yuan":28.48,"start":"2026-03-16T08:30:00","end":"2026-03-16T09:15:00","discount":true}'
```

预期输出：`✗ 验证失败`

**场景 4：删除字段**

```bash
# 删掉了 end 字段
node bin/energy-cli.js verify \
  -d '{"station_id":"CS-HZ-001","order_id":"ORD-20260316-001","user_id":"U10086","kwh":35.6,"amount_yuan":28.48,"start":"2026-03-16T08:30:00"}'
```

预期输出：`✗ 验证失败`

**场景 5：改变字段顺序**

```bash
# 把 station_id 移到最后
node bin/energy-cli.js verify \
  -d '{"order_id":"ORD-20260316-001","user_id":"U10086","kwh":35.6,"amount_yuan":28.48,"start":"2026-03-16T08:30:00","end":"2026-03-16T09:15:00","station_id":"CS-HZ-001"}'
```

预期输出：`✗ 验证失败`（JSON 字符串不同 → keccak256 哈希不同）

### 9.3 先算哈希再验证

```bash
# 第一步：计算原始数据的哈希
node bin/energy-cli.js hash \
  -d '{"station_id":"CS-HZ-001","order_id":"ORD-20260316-001","user_id":"U10086","kwh":35.6,"amount_yuan":28.48,"start":"2026-03-16T08:30:00","end":"2026-03-16T09:15:00"}'
# 输出: 0xabcdef1234...

# 第二步：用哈希在链上查验
node bin/energy-cli.js verify --hash 0xabcdef1234...
```

### 9.4 篡改检测原理

```
原始数据: {"amount":100}  →  keccak256  →  0xAABB...CC
篡改数据: {"amount":200}  →  keccak256  →  0xDDEE...FF  (完全不同！)

链上只有 0xAABB...CC 的记录 → 用 0xDDEE...FF 查询 → 找不到 → 验证失败
```

**结论**：任何 1 个字符的改动都会导致完全不同的哈希值，篡改立刻可检测。

## 十、批量数据上链脚本（xlsx 电表数据）

项目提供了两个脚本，分别用两种方式将 `docs/有功功率数据.xlsx`（3 个电表、约 9000 条有功功率记录）上链。

### 10.1 安装脚本依赖

```bash
cd scripts
npm install
```

### 10.2 Script A — EVM 合约存证

通过 `EnergyDataAttestation.sol` 合约的 `attest()` 方法逐条上链。数据哈希写入合约 storage，可通过合约方法查询。

```bash
cd scripts
node upload_evm_contract.js
```

- 使用 **validator 私钥**
- 每条记录：计算 keccak256(JSON) → 调用 `attest(hash, "active_power", memo)`
- Gas Limit: 600000（Cosmos EVM 实际消耗约 300k）
- 预计耗时：约 7 分钟（9000 条，2 秒出块）

### 10.3 Script B — EVM 原始交易存证

Script B 使用 **dev0 账户**，运行前需确保 dev0 有足够 ECY 支付 Gas。先从 validator 转入 1000 ECY：

```bash
# 从 validator 向 dev0 转入 1000 ECY
cd scripts
node --input-type=module -e "
import { ethers } from 'ethers';
const provider = new ethers.JsonRpcProvider('http://127.0.0.1:8545');
const validator = new ethers.Wallet('0x8be1e5311e4cb31002c5c84cea459b5e598592f1d00c796e3de2880d55fe9990', provider);
const dev0 = '0xC6Fe5D33615a1C52c08018c47E8Bc53646A0E101';
const tx = await validator.sendTransaction({ to: dev0, value: ethers.parseEther('1000') });
await tx.wait();
console.log('转账完成，dev0 余额:', ethers.formatEther(await provider.getBalance(dev0)), 'ECY');
"
```

确认 dev0 有余额后，运行上链脚本：

```bash
node upload_evm_rawtx.js
```

发送原始 EVM 自转交易，将数据哈希和元信息编码在交易的 `data` (calldata) 字段中。数据永久记录在区块中。

- 使用 **dev0 私钥**
- 每条记录：自转 0 ECY，calldata 包含 `{schema, category, data_hash, meter, time, value}`
- Gas Limit: 100000
- 预计耗时：约 5 分钟

### 10.4 两种方式对比

| 维度 | Script A（合约存证） | Script B（原始交易存证） |
|------|---------------------|------------------------|
| 存储位置 | 合约 storage（mapping） | 交易 calldata（区块数据） |
| 可查询性 | 合约函数查询 | 需按 txHash 或扫描区块 |
| Gas 成本 | 较高（~300k/笔） | 较低（~50k/笔） |
| 适用场景 | 需要链上索引和查询 | 只需证明数据上链 |
| 浏览器显示 | Blockscout 合约交互 | Blockscout 普通转账 |

## 十三、溯源原理说明

```
┌─────────────────────────────────────────────────────────────┐
│                       数据溯源流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【上链阶段】                                                │
│                                                             │
│   原始数据 ───── keccak256 ─────► 数据哈希 (32 bytes)        │
│   (JSON/文件)     (链下计算)        (唯一指纹)               │
│                                                             │
│                       │                                     │
│                       ▼                                     │
│              ┌────────────────┐                             │
```
